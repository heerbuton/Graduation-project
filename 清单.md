# 伯牙解谱（Boyajiepu）系统总进度与文件清单

这是一份关于“伯牙解谱”项目目前所有文件结构说明、开发进度以及您后续需要操作完善之待办事项的详细文档。

## 📂 第一部分：项目全量文件清单与作用梳理

整个项目主要分为前后端两个大独立服务模块，以及相关的项目文档。具体目录与文件说明如下：

### 1. 根目录基础文件
- **`README.md`**：项目使用说明书。包含了项目的基本介绍、架构结构概览、预留接口说明以及如何分别启动前端和后端的详细指南。
- **`todo.md`**：后端算法介入部分的后续待办事项。也是本清单中“欠缺操作部分”的直接来源和细分指导。

### 2. 后端服务部分（`backend/`）
后端基于 Flask 强力驱动，主要作用是接收前端的请求，调用算法流水线，最终反馈可渲染的 MusicXML 结构。
- **`app.py`**：后端核心路由应用入口文件。内部设置了与前端通信的跨域许可（CORS），以及包含 `/api/process_image`（真实算法调用）和 `/api/mock_pipeline`（测试通道）两大接口，同时负责异常时的防崩溃干预。
- **`best.pt`**：YOLOv8（或其他版本 YOLO）的核心模型权重文件，包含您事先训练好的对于古琴减字组件（偏旁、部首等）目标检测网络特征库。
- **`requirements.txt`**：Python 依赖清单。记录了后端项目运行所需的全部第三方库（如 `flask`, `flask-cors`, `ultralytics`, `requests` 等）。
- **`static/uploads/`**：用户上传图片时的缓存与落盘区，在服务端硬盘中保存实际的图片，避免了直接处理 Base64 带来的潜在数据乱码或异常。
- **`venv/`**：Python 本地虚拟环境目录。它保障了该项目依赖库的独立性，不会影响您系统上的其他 Python 项目环境。
- **`pipeline/`**：端到端核心调度流水线模块（**重点**）：
  - **`cv_module.py`**（模块A）：计算机视觉感知模块。负责去加载 `best.pt` 权重文件，并调用模型进行目标检测。
  - **`topology_module.py`**（模块B）：空间拓扑聚类模块。利用算法把 YOLO 识别出的零散部件（比如“大”、“九”、“勾”单独的框）合并为具有语义关联的一个具体古琴谱汉字组合。
  - **`llm_module.py`**（模块C）：大型语言模型（LLM）推理打谱模块。根据前一步聚类出的语义组件集合，编写提示词访问大语言模型（如 ChatGPT 或国内模型），完成核心的音高、时值推断并确定小节线生成 JSON 逻辑序列。
  - **`musicxml_encoder.py`**（模块D）：XML 构建生成模块。把 LLM 吐出来的逻辑数据重组封装，输出一段符合国际 MusicXML 规范且内嵌了减字详细分解部分的长字符串，以供前端渲染。

### 3. 前端部分（`frontend/`）
基于 Vue 3 + Vite 构建的高性能、高颜值的可视化交互界面。
- **`package.json` & `package-lock.json`**：Node.js 环境工程文件。包含了前端依赖树及其锁定版本，以及运行脚本定义（如 `npm run dev`）。
- **`index.html`**：前端应用的单页 HTML 入口，Vite 会将构建后的 Vue 代码注入至此。
- **`tailwind.config.js` & `postcss.config.js`**：Tailwind CSS 的核心配置文件，负责前端的响应式与原子化样式定义体系。
- **`vite.config.js`**：Vite 脚手架打包、开发服务器构建配置文件。
- **`src/`**：Vue 源码目录：
  - **`main.js`**：挂载整个 Vue app 及全局样式注册点。
  - **`index.css`**：全局基础样式池，包含 Tailwind 指令。
  - **`App.vue`**：系统的顶级布局（左侧图片上传展示器 + 右侧解析渲染结果展示区），控制全局状态和上传事件。
  - **`components/MusicXmlScoreRenderer.vue`**：★ **前端的最核心组件。完全脱离第三方庞大开源库引擎自研的渲染器。** 能够将后端的 XML 巧妙地利用 Flex 容器和 CSS 进行绝对精美、高颜值的水平+垂直（针对古琴多层叠写减字标注、减时线、小节线、附点体系等）双维度精准排版，最终实现乐谱的呈现。

### 4. 其他结构
- **`walkthroughs/`**：可用于存放渲染排版效果预览图（如 `final_rendered_result.png`）或过程展示文件资料的文件夹。

## 🛠️ 第三部分：当前欠缺的地方及您需要操作完善的步骤

当前系统的**瓶颈在后端核心算法替换**。系统目前是一个极其好看且能跑通的**“空壳骨架体验版”**，其算法预测返回均来自写死的 Mock 字典，尚未实现遇到不同的真实图片就跑不同的动态打谱。

**为了让系统成为能投入使用和答辩的“完全体”，您需要亲自完成下列编码和模型的嵌入式对接：**
*(所有操作均在 `backend/pipeline/` 目录下)*

### 任务 1：注入您的视觉之眼（YOLO 替换）
由于图像检测权重不能写在代码里，所以：
1. 请确保把您自己训练好的真实模型权重文件复制覆盖到 `backend/best.pt` 或者对应路径。
2. 进入 `backend/pipeline/cv_module.py` 文件内部。
3. 把文件当前开头直接返回我写的字典的数据抹除，通过 `from ultralytics import YOLO`（确保您通过 pip 装好了该库），然后使用 `model = YOLO('best.pt')` 载入。
4. 在 `detect_components(image_path)` 函数里执行真实的 `model.predict(image_path)`，将该调用的返回信息中提取出边框坐标 `[x1, y1, x2, y2]` 和文本分类 `class`，转换成代码原来要求的那个 List of Dict 的标准 JSON 返回格式返回！

### 任务 2：开发您的空间聚类算法（Topology 定制）
目前 YOLO 扫出来的可能只是“大”、“九”、“勾”单独的框。我们需要组合。
1. 进入 `backend/pipeline/topology_module.py`。
2. 里面现存的同样是假数据。您需要根据自己的课题所研究的方向，写一个空间匹配合并代码：如果某几个部件它们中心点的欧拉距离够近、并且从属结构合理，那么就把这几块判定为构成一个整体部件（聚类）。
3. 将聚散的框归纳成由多个部件连写在一起的一个“群组”（代表一个完整的音符单位），再封装输出。

### 任务 3：让系统拥有真正的智慧（接入 LLM 大语言模型 API）
1. 在 `backend/pipeline/llm_module.py` 中，你需要准备任一提供商模型的真实 API Key (比如通义千问、DeepSeek、GPT-4 或者智谱调用密钥)。
2. 通过 Python 的 `requests` 模块向真实接口发起流式或者块调用请求。
3. **编写极为关键的核心 Prompt（提示词）**：您需要根据拓扑整理出的文字部件集合，向模型提问：“在这个场景下正调打谱，上述减字符号的音高是什么，时值到底是多少？”
4. 强制要求大模型吐出“仅包含 JSON”格式的结构，因为你要把获取结果使用 `json.loads` 转成我们后面音乐渲染器要认的严格数据结构。

### 任务 4：（拓展建议）格式强校验
在完成打谱逻辑后即可基本结束项目。不过如果您想锦上添花：
可以在 `backend/pipeline/musicxml_encoder.py` 末尾生成的 XML 上下文，加入 `lxml` 库的解析，并导入真正的 MusicXML 标准文档（XSD格式），确保您那端模型偶然乱输出的情况下、生成并不会直接发往前台导致前台画面炸裂崩溃而是提前进行拦截报错。

***

**结语**：
只要顺着以上四个步骤，将您自己跑过并验证好的模型脚本“套”入现成的四个 `pipeline.py` 的管内（填空题），整个端到端原型就能实现真实环境中的完整全链路动态打谱展示了！祝您答辩顺利完成出色的演示。
